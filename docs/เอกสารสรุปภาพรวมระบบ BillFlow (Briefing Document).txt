เอกสารสรุปภาพรวมระบบ BillFlow (Briefing Document)
บทสรุปผู้บริหาร (Executive Summary)
BillFlow คือระบบบริหารจัดการใบสั่งซื้อและบิลแบบรวมศูนย์ที่ออกแบบมาเพื่อลดระยะเวลาการทำงานของเจ้าหน้าที่จากเดิมที่ต้องคีย์ข้อมูลด้วยตนเอง (Manual Entry) ประมาณ 100 บิลต่อวัน ให้เหลือเข้าใกล้ศูนย์ ระบบทำงานโดยการรวบรวมข้อมูลจากช่องทางที่หลากหลาย ได้แก่ LINE Official Account (OA), Email (IMAP), และการนำเข้าไฟล์ Excel จาก Shopee โดยเปลี่ยนกระบวนการจากการคีย์ข้อมูลเป็นการตรวจสอบและยืนยัน (Manual-confirm flow) ผ่าน AI และระบบอัตโนมัติก่อนส่งข้อมูลเข้าสู่ระบบ ERP SML 248ประเด็นสำคัญของระบบ:
* การจัดการหลายช่องทาง:  รองรับ LINE OA และบัญชีอีเมลหลายบัญชีพร้อมกัน พร้อมระบบ Routing อัตโนมัติตามเนื้อหาหรือหัวข้ออีเมล
* ความแม่นยำของข้อมูล:  ใช้ AI ในการสกัดข้อมูลจาก PDF, รูปภาพ และเสียง พร้อมระบบ Item Mapping (F1) เพื่อจับคู่ชื่อสินค้าที่ลูกค้าพิมพ์มากับรหัสสินค้าใน ERP
* ความปลอดภัยและความถูกต้อง:  มีระบบ Validation Guard ป้องกันการส่งบิลที่ข้อมูลไม่ครบถ้วน และใช้ระบบ Manual-confirm เพื่อป้องกันข้อผิดพลาดก่อนเข้าสู่ ERP
* ประสิทธิภาพการทำงานสูง:  รองรับการอัปเดตสถานะแบบ Real-time ผ่าน Server-Sent Events (SSE) และมีระบบ Hybrid Reply/Push เพื่อบริหารจัดการโควต้าข้อความ LINE อย่างคุ้มค่า
1. สถาปัตยกรรมและการไหลของข้อมูล (Architecture & Data Flow)
ระบบเปลี่ยนจากรูปแบบการส่งข้อมูลอัตโนมัติ (Auto-send) มาเป็น  Manual-confirm flow  ตั้งแต่ Session 5 เป็นต้นไป เพื่อให้มั่นใจในความถูกต้องของข้อมูลใน ERP โดยมีลำดับขั้นตอนดังนี้:
1. Input:  รับข้อมูลจาก LINE (ข้อความ/รูปภาพ/ไฟล์), Email (PDF/Shopee Order), หรือ Shopee Excel
2. Process:  ระบบทำการ Parse ข้อมูล ตรวจสอบรายการซ้ำ (Dedup) และใช้ AI Extract ข้อมูลสำคัญ (กรณีเป็นสื่อหรือข้อความที่ไม่มีโครงสร้าง)
3. Validation:  Admin ตรวจสอบความถูกต้องที่หน้า /bills/:id โดยมี Validation Guard คอยตรวจสอบความสมบูรณ์ของข้อมูลรายการสินค้า
4. Output:  ส่งข้อมูลไปยัง SML ERP ผ่าน API 248 โดยมีการจัดการรูปแบบวันที่และเลขที่เอกสารตามมาตรฐานบัญชีไทย
2. การจัดการช่องทางอีเมล (Email IMAP Management)
ระบบรองรับการเชื่อมต่อกับ Inbox หลายบัญชีพร้อมกัน โดยแต่ละบัญชีจะถูกแยกการทำงานผ่าน Goroutine และมีการตั้งค่าแยกกันผ่าน UI /settings/email
ฟีเจอร์หลักและการตั้งค่า:
* Per-channel Routing:
* general: สำหรับ PDF บิลทั่วไป ระบบจะใช้ AI Extract ข้อมูลเหมือนกับช่องทาง LINE
* shopee: สำหรับอีเมลจาก Shopee โดยจะแยกตาม Keyword ในหัวข้อ (สั่งซื้อ, จัดส่ง, ชำระเงิน) เพื่อส่งไปยัง Route ที่ถูกต้องใน SML (Sale Order หรือ Purchase Order)
* Dedup Logic:  ใช้ Message-ID ตามมาตรฐาน RFC 5322 เพื่อป้องกันการประมวลผลอีเมลซ้ำ (Skip หากเคยประมวลผลแล้ว)
* Admin UI:  รองรับการทำ Test Connection, Enumerate Mailboxes และ Manual Poll (ดึงข้อมูลทันทีโดยไม่ต้องรอรอบ Interval 5 นาที)
3. ระบบ LINE OA และ Chat Inbox
BillFlow ทำงานเป็น Inbox สองทางสำหรับ LINE OA หลายบัญชี โดยเน้นการบริหารจัดการผ่าน Admin และลบระบบ Chatbot อัตโนมัติออกเพื่อความแม่นยำ (Session 13)
รายละเอียดการทำงานของ LINE OA:
คุณสมบัติ,คำอธิบาย
Hybrid Reply/Push,ใช้ Reply API เป็นอันดับแรก (ฟรี) และใช้ Push API เมื่อจำเป็น (โควต้า 200 ครั้ง/เดือน)
Real-time SSE,อัปเดตข้อความใน Inbox ทันที (Sub-second) โดยแสดงสถานะการเชื่อมต่อ (Live/Offline) ที่ Sidebar
CRM Lite,"ระบบตรวจจับเบอร์โทรศัพท์อัตโนมัติ, การบันทึก Note ภายใน และการติด Tag (7 สี) เพื่อคัดกรอง Inbox"
Media Extraction,"Admin สามารถคลิก ""🔍 สร้างบิลจากสื่อนี้"" เพื่อให้ AI ช่วยดึงข้อมูลจากรูปภาพ, PDF หรือเสียง"
4. การนำเข้าข้อมูล Shopee Excel (Shopee Import)
พนักงานสามารถ Export ไฟล์จาก Shopee Seller Center และนำเข้ามาที่ระบบเพื่อแปลงเป็นบิลได้อย่างรวดเร็ว
* Column Mapping:  ระบบใช้การดึงข้อมูลจากชื่อคอลัมน์ภาษาไทยที่คงที่ (เช่น หมายเลขคำสั่งซื้อ, ชื่อสินค้า, SKU)
* Excluded Statuses:  ระบบจะข้ามคำสั่งซื้อที่มีสถานะ "ที่ต้องจัดส่ง" และ "ยกเลิกแล้ว" โดยอัตโนมัติ
* SKU Lookup:  มีการจัดลำดับความสำคัญ (Priority) ในการหารหัสหน่วยสินค้า (Unit Code), คลัง (WH), และที่เก็บ (Shelf) หากไม่พบจะใช้ค่า Default จากการตั้งค่าระบบ
5. การเชื่อมต่อ SML ERP 248 และความถูกต้องของข้อมูล
การส่งข้อมูลเข้าสู่ SML ERP ได้รับการปรับปรุงให้สอดคล้องกับมาตรฐานบัญชีและแก้ไขปัญหาทางเทคนิคที่เคยพบ
การจัดการวันที่ (Doc Date Logic):
ระบบแยก Concept ของวันที่ออกเป็น 3 ส่วน เพื่อความถูกต้องของรายงานใน ERP:
* doc_date:  วันที่ปัจจุบันที่กดส่งข้อมูล (Today)
* doc_ref:  หมายเลขอ้างอิงจากแหล่งที่มา (เช่น Shopee Order ID)
* doc_ref_date:  วันที่ที่เกิดเอกสารจริงจากแหล่งที่มา (เช่น วันที่สั่งซื้อใน Shopee)
มาตรการป้องกันความผิดพลาด:
* Validation Guard:  ปุ่ม "ส่ง SML" จะถูกระงับ (Disabled) หากข้อมูลรายการสินค้าไม่ครบถ้วน เช่น ไม่มีรหัสสินค้า, จำนวนน้อยกว่าหรือเท่ากับ 0, หรือไม่มีหน่วยสินค้า
* MarshalASCII Workaround:  ระบบใช้ฟังก์ชันพิเศษในการจัดการ Body ของ JSON เพื่อแก้ปัญหาตัวอักษรไทยอ่านไม่ได้ (Mojibake) บน SML Server ที่ไม่รองรับ UTF-8 ในบางกรณี
* Retry Policy:  พยายามส่งซ้ำสูงสุด 3 ครั้งพร้อมระบบ Backoff (0s, 1s, 3s, 5s) และคงเลขที่เอกสารเดิมไว้เพื่อป้องกันการรันเลขที่ผิดพลาด
6. งานระบบและการตั้งค่า (System Admin & Jobs)
ระบบย้ายการตั้งค่าส่วนใหญ่จาก .env มาไว้ใน Database เพื่อให้ Admin จัดการผ่านหน้าเว็บได้โดยตรง
ตารางงานพื้นฐาน (Background Jobs):
* Email Coordinator:  ดึงอีเมลใหม่ทุกๆ 5 นาที (หรือตามที่ตั้งค่า)
* Daily Insight (F4):  AI สรุปยอดขายรายวันและแจ้งเตือนผ่าน LINE Admin เวลา 08:00 น.
* Tunnel Drift Monitor:  ตรวจสอบความสมบูรณ์ของ Public URL (Cloudflare) ทุกวันเวลา 09:00 น.
* Maintenance:  ระบบสำรองข้อมูล (Backup) เวลาเที่ยงคืน และตรวจสอบวันหมดอายุของ LINE Token ทุกวันจันทร์
บันทึกการตรวจสอบ (Audit Log):
ทุกการกระทำที่สำคัญจะถูกบันทึกไว้ในระบบเพื่อตรวจสอบย้อนหลังได้ ได้แก่:
* การดึงข้อมูลอีเมลสำเร็จ/ล้มเหลว
* การสร้างบิลจากช่องทางต่างๆ
* การส่งข้อมูลเข้า SML (สำเร็จ/ล้มเหลว พร้อมรายละเอียด Error)
* การแก้ไขสถานะการสนทนาใน LINE
